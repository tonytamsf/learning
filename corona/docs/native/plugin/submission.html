Plugin Submission Guide

Revision Current Public Release (2013.2076)
Keywords plugin, marketplace
See also Marketplace

Overview

This guide is for 3rd parties who are ready to submit a plugin.

Distribution

Plugins are distributed differently depending on the product;

  • Corona SDK
    • Your plugins will be hosted live on Corona's build servers, so your plugins are available immediately to all SDK developers.
    • When a SDK developer creates a project that requires your plugin, the build server automatically integrates the plugin during the build process.
  • Corona Enterprise
    • The Enterprise developer will have to include the plugin into the native project.
    • Depending on the target OS, this can take several forms: static library (iOS), shared library (Mac/Win/Android), or JAR file (Android).

Packaging for Submission

Plugins should be submitted via a bitbucket repo that has been shared with you. The structure of that repo must follow a particular format.

When you have an update to your repo, you can generate a pull request.

To help you get started, we have created a Plugin Submission Template that contains starter files, with the exception of the plugins binary directory.

Directory Structure

You should package up your files in the following format

  • metadata.json Key information about your plugin. See 'Format for metadata.json' below.
  • docs/ Contains documentation. See 'Creating Documentation' below.
  • plugins/
    • ${VERSION}/ Daily build version, e.g. 2013.1076. See 'Plugin Versioning' below.
      • android/
        • Binary (.jar) (.so support coming soon)
        • metadata.lua See 'Format for per-platform metadata.lua' below.
      • iphone/
        • Binary (.a) (armv7)
        • metadata.lua See 'Format for per-platform metadata.lua' below.
      • iphone-sim/
        • Binary (.a) (i386)
        • metadata.lua See 'Format for per-platform metadata.lua' below.
      • mac-sim/
        • Binary (.dylib) or Lua plugin
      • win32-sim/
        • Binary (.dll) or Lua plugin
  • samples/ This contains the SDK-style project, e.g. 'main.lua', that we can use to verify your plugin.

Format for metadata.json

This json file contains information about the publisher, the name of the plugin as it appears in Lua, and other related information.

The Plugin Submission Template contains a metadata.json file that should be updated with the appropriate information.

Here's an example:

{
    "contact": "john@mycompany.com",
    "url": "http://www.mycompany.com",
    "pluginName": "plugin.mylibrary",
    "publisherId": "com.mycompany"
}

Format for per-platform metadata.lua

The metadata.lua file describes properties of the plugin binary. This file is required for the following target platforms:

  • Android
  • iOS
  • iOS Simulator

The Plugin Submission Template provides stub versions. Examples are listed below:

Android

On Android, you need to specify the format of the binary. Currently .jar is supported. Support for native .so shared libraries is coming soon.

In addition, you can specify elements in the AndroidManifest.xml file that need to be modified in order to accommodate the plugin.

local metadata =
{
    plugin =
    {
        format = 'jar',
        manifest = 
        {
            permissions = {},
            usesPermissions =
            {
                "android.permission.INTERNET",
                "android.permission.ACCESS_NETWORK_STATE",
                "android.permission.READ_PHONE_STATE",
            },
            usesFeatures = {},
            applicationChildElements =
            {
                -- Array of strings
                [[
<activity android:name="com.mycompany.MyActivity"
          android:configChanges="keyboard|keyboardHidden"/>]],
            },
        },
    },
}

return metadata

iOS and iOS Simulator

The iOS and iOS Simulator metadata.lua files should be the same.

In both, you specify the format of the binary. Currently only static libraries are supported.

For frameworks, Corona already links to a list by default (see 'Default Frameworks' below). If you need to link against additional frameworks, you can specify them in the metadata.lua file as either required frameworks or optional frameworks (weak linking). When you list them, please remove the .framework extension.

Here's an example:

local metadata =
{
    plugin =
    {
        format = 'staticLibrary',

        -- This is the name without the 'lib' prefix.
        -- In this case, the static library is called: libplugin_openudid.a
        staticLibs = { 'plugin_openudid', },

        frameworks = {},
        frameworksOptional = {},
    },
}

return metadata

Creating Documentation

These files should be written in markdown files. Once your submission is accepted, these markdown files will be converted and hosted on our docs site.

Your markdown files should follow the markdown files in the Plugin Submission Template. There is more information on how to deal with linking between multiple markdown files and what flavor of markdown we support in the template Readme.

Plugin Versioning

When a plugin is published to Corona's servers, you must specify a daily build version for the plugin binary you are deploying.

This daily build version controls the availability of your plugin. Specifically, you will need to use that version (or subsequent versions) of the Corona SDK in order to use that plugin in a device build and, when available, in the Corona Simulator.

For example, if you want your plugin to be available starting in daily build 2013.1076, then you should use that for ${VERSION} per the above directory structure.

Then you would tell developers to install Corona SDK version 2013.1076 in order to access that plugin.

Choosing a Version

For new plugins, we recommend you use the most recent public release as your version. This ensures that your plugin is available to the broadest audience possible.

When you build your plugin binary, you should use the corresponding version of CoronaEnterprise. For example, if you choose daily build version 2013.1076, then you should build using CoronaEnterprise 2013.1076.

Multiple Versions

Normally, when you update a plugin binary, you just resubmit the plugin.However, there may be an occasion in which a newer version of the plugin breaks backwards compatibility (e.g. API change) or your plugin depends on new functionality available in a later version of Corona.

In this situation, you should deploy a separate plugin binary. This binary will correspond to a different/newer daily build version of Corona SDK. A developer then simply grabs that newer Corona SDK daily build to access the new version of the plugin. In this way, you can keep the legacy plugin binary around to preserve backwards compatibility.

For example, here's what your plugins directory might look like:

  • plugins/
    • 2013.1076/ Contains the older legacy plugin binaries.
    • 2013.1100/ Contains the new plugin binaries (e.g. different API)

FAQ

Do I need to create plugins for platforms that are not supported, such as Windows and OSX?

At a minimum you will need to supply a LUA plugin stub for all platforms (including plugin versions for Windows and OSX). These stubs should allow Corona SDK developers to make calls as they would on a fully supported platform. A sample LUA stub can be found here: https://github.com/coronalabs/plugins-lua-template/blob/master/shared/plugin_PLUGIN_NAME.lua

I have uploaded a plugin for 'platform X'. Where do I add plugins for the additional platforms?

Under the daily build version of the plugin folder you should supply your plugin implementations within the folders named android, iphone, iphone-sim, mac-sim, and win32-sim.

I have created static libraries for iOS (.a files) and jar files for Android (.jar files for Android), how do I produce dll's and dylibs for the simulators?

If the plugin does not have an implementation for Windows or OSX then we recommend creating LUA stub files (see above.)

If you wish to provide Windows or Mac simulator plugin implementations, you will need to use Visual Studio and Xcode respectively, to produce Dll's and Dylibs. A few links of interest can be found here:

Visual Studio DLL Walkthrough

Xcode Dylib Walkthrough

We have updated our documentation and/or libraries, but they are not available?

Please contact us whenever there is a change made to the documentation or libraries to make sure that we are aware of your change and that you wish to make it publicly available. plugins@coronalabs.com.

Why is there an iphone and iphone-sim folder?

When you submit a plugin, two versions should be submitted. The iphone folder should contain the armv7 plugin and the iphone-sim folder should contain the i386 version of the plugin. This ensures that there is a plugin available for both the XCode simulator and the physical device.

Can I include resources in my Android plugin?

Resources cannot be included directly in an Android plugin, it's recommended that the plugin download resource files it requires.

Which version of the Android NDK should I be using for native library (.so) plugin creation**

For plugins available for Corona SDK builds 2013.1148 and above the native library (.so) plugin should be built with the Android NDK labelled android-ndk-r8b.

Where should I post my plugin sample code?

We recommend uploading sample code to a GitHub repo and linking to it from within the plugin index.markdown under the header "Sample Code".

How can I test my plugin?

The plugin can be tested using the Enterprise version of Corona SDK by linking the static library (iOS) or through including the library in your project's libs directory (Android). After initial testing with Enterprise, you will need to submit the binaries to Corona Labs to test with the Pro version of Corona SDK.
