Display Groups and Coordinates

This guide covers the Corona coordinate system and how to position objects on the screen. Additionally, you'll learn about display groups and how they relate to Corona's "drawing model".

  • Cartesian Coordinate System
  • Corona's Stage and Coordinates
  • Corona Display Groups

Cartesian Coordinate System

A Cartesian coordinate system (also known as a rectangular coordinate system) is generally used to define position. In this system, you have x values and y values — x defining the "horizontal" position and y defining the "vertical" position.

If you're not familiar with basic Cartesian coordinates, a convenient context is to imagine a map (of a country, a state, a city, etc.). On this map, the specific location of a landmark is defined by its x and y coordinates, expressed as (x,y). So, imagine a map of North America: the city of New York might be located at (100,20), while the city of Los Angeles might be located at (-120,80). As you can see, coordinates can be expressed by negative values as well as positive values.

Corona's Stage and Coordinates

In Corona, the screen — or stage, as it's frequently referred to — can likewise be imagined as a map. Objects that you place on the stage will have x and y coordinates indicating their horizontal and vertical position, just as the city of New York has specific coordinates in the example above.

In standard Cartesian coordinates, the exact center of the map (its "origin") is (0,0). Coordinates to the left of horizontal center are represented by negative x values, while those to the right of horizontal center are represented by positive x values. Coordinates above the vertical center are represented by positive y values, and coordinates below the vertical center are negative y values.

This typical system is "shifted" in Corona. In Corona, the origin of the screen (0,0) is located at the top-left corner of the stage, not the center! Additionally, positive y coordinates extend downward instead of upward; however, positive x values still extend to the right, as usual.

And what about negative coordinates in Corona? They are still allowed, but they are not within the visible bounds of the stage. Negative x coordinates reside off the left edge of the screen and negative y values reside off the top edge.

Corona Display Groups

Understanding Corona display groups is essential to building apps in the SDK. A display group is actually a special kind of display object which can contain other display objects and even other display groups. Imagine it as a blank sheet of paper on which you'll eventually "draw" images, text, shapes, and animated sprites (see the Display Objects guide for more info).

While a particular display group generally represents the edge-to-edge boundaries of the screen with coordinates (0,0) at the top left, it's not limited to those boundaries. In fact, a display group spans an infinite amount in all four directions — the physical screen edges simply frame a portion of the display group.

Let's emphasize the "sheet of paper" analogy. In actual practice, you will place display objects in a display group. Just as if you're drawing sketches on a sheet of paper, Corona display objects placed in a display group become part of that group.

Note

A display group is itself a display object. Therefore, it obeys all of the properties that are common to other display objects such as x position, y position, rotation, and alpha. This means that you can use groups to manipulate related sets of display objects. For example, if you create a menu with several buttons, you should place the buttons inside a group and then move/position/rotate/fade the group as an entire unit, directly or with a transition.

"Stack of Paper"

Now let's expand on the analogy. In the process of developing an app, you will most likely use several display groups in a specific layered order. You can imagine this as a stack of paper with each display group representing one sheet in the stack. For example, you might want to mimic a "landscape painting" in Corona with three layers (three display groups).

  1. far background — contains sky, clouds, etc.
  2. near background — contains background mountains
  3. foreground — contains trees, grass, rocks, etc.

Each display group would contain the appropriate display objects, thus composing the painting in the proper layered order. Of course, while each layer represents a sheet of paper in our analogy, you can't actually see the "paper" itself (it will not appear white). Just imagine an invisible sheet on which you place your text, images, etc.

Creating a Display Group

Creating a new display group is as simple as the following:

local myGroup = display.newGroup()

This creates a new display group by the name of myGroup. You can then add display objects to it. Because you will refer to and access display groups throughout your code, you should declare them near the top of your Lua document. This ensures that other APIs recognize the names and layered order of all display groups.

Important

You should create display groups in a specific order. The first declared group will actually reside behind the next group in visually layered order. In the "stack of paper" analogy, the sheet (display group) on the bottom of the stack should actually be declared first in your code, and you should work upward to the top of the stack. For example, to create the "landscape painting" in the example above, declare your three display groups as follows:

local farBackground = display.newGroup()  
local nearBackground = display.newGroup()  --this will overlay 'farBackground'  
local foreground = display.newGroup()  --and this will overlay 'nearBackground'

Stage = Core Display Group

Corona's stage, discussed above, is actually a display group too, but you don't need to explicitly declare it in your code. The stage can be considered the "core display group," and any custom display groups that you declare are automatically added as child groups of the stage. You can, of course, add display objects directly to the stage, but we recommend that you create a hierarchy of ordered display groups. Doing so will make it easier to manage your display objects, move them from one display group to another, change their front-to-back order compared to other objects in the same group, etc. Most apps utilize a relational hierarchy of display groups to organize display objects, so it's considered "best practice" to begin with a basic foundation.

Notes

  • Although display groups are essentially tables, Lua library functions like table.insert() and ipairs() are not compatible with groups. In addition, you cannot get the number of children in a display group using #myGroup. Instead, use myGroup.numChildren.

  • Both display.getCurrentStage() and display.currentStage return a reference to the current stage object.

  • More details about display objects and display groups are contained in the Display Objects guide.
